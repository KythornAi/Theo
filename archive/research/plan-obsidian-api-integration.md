# Plan: Obsidian Local REST API Integration for Theo

**Date:** 2026-04-29
**Status:** Draft — awaiting Kyle + Claude review
**Risk level:** Low (read-only operations only, no vault mutations initially)

---

## Problem

- Theo's existing Obsidian skill is file-based only (`cat`, `find`, `grep`)
- No structured API, no safety checks, no frontmatter/sub-document targeting
- Need proper read/write capability for research notes, task tracking, and knowledge management
- Must not compromise Obsidian vault integrity or other users' workflows

## Solution

Use the **obsidian-local-rest-api** plugin to give Theo a secure, authenticated HTTP interface into the vault.

---

## How the Plugin Works

- **Serves on `https://127.0.0.1:27124`** (default port, configurable)
- **Self-signed HTTPS certificate** — API key auth via `Bearer` token in `Authorization` header
- **Requires Obsidian desktop app to be running** — it's an in-app plugin, not a standalone daemon
- **Full CRUD on vault files** — read, create, update, delete, append, partial patch

## Key Capabilities We Need

| Capability | Endpoint | What it gives us |
|-----------|----------|-----------------|
| Read any note | `GET /vault/{path}` | Get note content as JSON or raw markdown |
| List vault contents | `GET /vault/` or `/vault/{dir}/` | Discover structure, find notes |
| Create note | `PUT /vault/{path}` | Write new notes programmatically |
| Append to note | `POST /vault/{path}` | Append content without reading first |
| Update note | `PUT /vault/{path}` or `PATCH /vault/{path}` | Partial updates, section-level edits |
| Search notes | `POST /search/simple/` | Full-text fuzzy search with context snippets |
| Tags | `GET /tags/` | List all tags and usage counts |
| Status check | `GET /` | Verify plugin is running and auth is valid |

## Security Model

- API key is generated by the plugin inside Obsidian → **vault owner controls access**
- Key is stored in `~/.hermes/.env` as `OBSIDIAN_API_KEY` — never in code or logs
- HTTPS with self-signed cert — Theo's `curl` calls use `-k` (skip cert verify) OR system installs the cert as trusted
- **Localhost only** — the API binds to `127.0.0.1`, no external network exposure by default
- No port forwarding or tunneling needed — Theo and Obsidian run on the same machine

---

## Potential Risks & Mitigations

| Risk | Mitigation |
|------|-----------|
| **Vault corruption if write logic bugs** | Phase 1: read-only operations only. No writes approved in initial rollout |
| **Obsidian app not running** | Plugin only works when Obsidian desktop is open. Theo should handle this gracefully: try API → if connection refused → report "Obsidian not running" instead of silent failure |
| **API key leakage** | Store only in `~/.hermes/.env`. Never echo, log, or include in messages. Use `curl` with header, not URL params |
| **Conflicts with Obsidian Sync** | The plugin only writes to local vault files. Obsidian Sync handles sync separately. Standard file-write conflict behavior applies |
| **Accidental deletion** | `DELETE` endpoint disabled initially. Only `GET`, `POST` (append), and `PUT` (create with new path) enabled in skill |
| **Performance** | Search on large vaults can be slow. Add timeout (10s) to curl calls. Use `grep` as fallback for quick lookups |

## Implementation Phases

### Phase 1: Setup & Read-Only Access (Safe)

1. Install **obsidian-local-rest-api** plugin in Obsidian (via Community Plugins → Browse → search "Local REST API" → Install → Enable)
2. Generate API key in Settings → Local REST API
3. Store API key in `~/.hermes/.env`: `OBSIDIAN_API_KEY=<key>` (and `OBSIDIAN_API_URL=https://127.0.0.1:27124`)
4. Test connectivity: `curl -k -H "Authorization: Bearer $OBSIDIAN_API_KEY" $OBSIDIAN_API_URL/`
5. Update Theo's `obsidian` skill to use the API for **read operations only**:
   ```bash
   # Read note
   curl -k -s -H "Authorization: Bearer ${OBSIDIAN_API_KEY:-}" \
     "${OBSIDIAN_API_URL:-https://127.0.0.1:27124}/vault/${1}"
   
   # Search notes  
   curl -k -s -X POST -H "Authorization: Bearer ${OBSIDIAN_API_KEY:-}" \
     -H "Content-Type: application/json" \
     -d "{\"query\":\"${query}\"}" \
     "${OBSIDIAN_API_URL:-https://127.0.0.1:27124}/search/simple/"
   
   # List vault structure
   curl -k -s -H "Authorization: Bearer ${OBSIDIAN_API_KEY:-}" \
     "${OBSIDIAN_API_URL:-https://127.0.0.1:27124}/vault/"
   ```
   - Keep file-based fallback (`grep`, `find`) if API is unavailable
   - Skill handles missing obsidian app gracefully

### Phase 2: Write Access (after Phase 1 confirmed stable)

6. Add **append** capability (`POST /vault/{path}`) — for adding research notes to the vault
7. Add **create** capability (`PUT /vault/{path}`) — for new notes only (no overwrite of existing)
8. Add **search with context** — use Dataview DQL queries if Dataview plugin is installed

### Phase 3: Rich Operations (optional, future)

9. **Sub-document targeting** — edit specific headings/sections without touching the rest of the note
10. **Periodic notes** — read/write daily notes, weekly reviews (`/periodic/daily/`)
11. **Tag management** — use `GET /tags/` to understand vault taxonomy
12. **Active file awareness** — read what's open in the Obsidian UI (useful for context)

## Skill Update (Draft)

After Phase 1 is confirmed, the `obsidian` skill will be updated to:

### Read Note
```bash
curl -k -s -H "Authorization: Bearer ${OBSIDIAN_API_KEY}" \
  "${OBSIDIAN_API_URL}/vault/${NOTE_PATH}" \
  -H "Accept: application/vnd.olrapi.note+json"
```
- Returns note as JSON with content, metadata, and frontmatter
- Falls back to `cat` if API unavailable

### Search Notes
```bash
curl -k -s -X POST -H "Authorization: Bearer ${OBSIDIAN_API_KEY}" \
  -H "Content-Type: application/json" \
  -d "{\"query\":\"${QUERY}\"}" \
  "${OBSIDIAN_API_URL}/search/simple/"
```
- Returns scored context snippets
- Falls back to `grep -rli` if API unavailable

### List Vault Structure
```bash
curl -k -s -H "Authorization: Bearer ${OBSIDIAN_API_KEY}" \
  "${OBSIDIAN_API_URL}/vault/"
```
- Returns JSON file tree

### Safety Rules
- **Never delete** (DELETE endpoint not used)
- **Create only** — new paths only, no overwrite existing (PUT only to non-existent paths)
- **Append only** — POST appends, never replaces

## Open Questions

- **Where does Obsidian run?** If the vault is on the Pi but Obsidian desktop app runs elsewhere (e.g., your Mac), the Local REST API only binds to localhost. Theo wouldn't be able to reach it without port forwarding or SSH tunneling. If vault is on Pi and Obsidian is on Mac, we need either a tunnel or an alternative approach (direct file access, which is what we already have with the current skill).
- **Does Obsidian actually run on Pi 5?** Obsidian doesn't have an ARM Linux build. If you're using it via Waydroid or similar, the REST API plugin may not work. We need to confirm how Obsidian runs in this setup before committing to this API approach.
- **Is the vault local or remote?** If vault is synced via Obsidian Sync, the local vault on Pi might just be a copy. Need to know the sync model.
